# Eval Roadmap

현재 평가 체계의 진행 상태와 보류 항목을 정리한다.

## 기본 명령

- 기본 진입 명령은 `yarn verify:fast`
- `yarn verify:docs`는 문서 계약 점검용 하위 명령
- `yarn test`는 기본 테스트만 보고 싶을 때 쓰는 하위 명령
- E2E 계열 명령은 기본 경로가 아닌 선택 명령

## 완료

- 기본 검증 경로를 `type -> test`로 정리
- 기본 테스트 러너를 `Jest`로 명시
- 문서 계약 검증을 `verify:docs`로 자동화
- 현재 service/controller 테스트와 잔존 E2E 범위를 인벤토리화

## 진행 중

- 남은 service branch, authorization rule, payload mapping을 계속 `test`로 이동
- 공백 문서와 인벤토리 문서의 갱신 누락을 줄이는 운영 습관 정착

## 보류

- DB fixture 기반 통합 테스트 확대
- S3 mock 서버 기반 통합 테스트
- rubric 기반 평가 계층 추가

## 판단 기준

### DB fixture 기반 통합 테스트

- 현재는 보류한다.
- 이유는 DB 연결이 필요한 케이스와 service 단위에서 검증 가능한 케이스를 먼저 분리해야 하기 때문이다.
- 인증/권한 회귀가 단위 테스트만으로 반복해서 누락될 때 다시 검토한다.

### S3 통합 테스트

- 현재는 외부 S3 직접 호출 테스트를 추가하지 않는다.
- key 생성, payload mapping, 에러 변환은 우선 순수 helper 또는 service test로 옮긴다.
- 실제 스토리지 권한 검증이 필요할 때 mock 서버 또는 전용 테스트 버킷 전략을 별도로 정한다.

### 문서 계약 자동화 확장

- 현재는 필수 문서 존재, 링크 존재, 기본 명령 일관성 정도만 확인한다.
- 내용의 적절성까지 기계적으로 강제하는 단계는 아직 보류한다.
- 문서 검증은 운영 계약을 보호하는 최소 수준에서 유지한다.

## 다음 작업 순서

1. auth service와 JWT strategy의 실패 분기를 `test`로 옮긴다.
2. S3 업로드 helper의 순수 로직을 분리해 `test`로 옮긴다.
3. DB 연결이 꼭 필요한 케이스만 E2E 후보로 남긴다.

## E2E 승격 기록 규칙

- 새 E2E를 추가하거나 기존 `test` 범위를 E2E로 올리려면 이 문서에 근거를 남긴다.
- 근거 없이 E2E를 늘리지 않는다.
- 근거에는 아래 중 어떤 이유인지 포함한다.
  - 실제 인증 연결 검증
  - 실제 DB 연결 검증
  - 실제 HTTP lifecycle 검증
  - 외부 스토리지 연동 검증
